# 智能摘要

把一份长内容压缩成能快速读懂的摘要。核心不是简单缩短，而是**在保真的前提下压缩**：判断内容是什么类型、按需要的详略档位输出、保留关键事实和行动项、绝不添加原文没有的信息。

## 目标

针对文章、会议纪要、代码、长对话等不同类型的内容，产出准确、简洁、结构合适的摘要。合格的摘要满足三条：**忠实**（不臆造、不曲解，观点归属清晰）、**有取舍**（保留主干和关键结论，删掉冗余铺垫）、**可用**（读者读摘要就能做出"是否需要读原文"的判断，或直接拿到需要的行动项）。

## 工作流步骤

### 步骤 1：获取内容并确认需求

- **来源**：用户可能粘贴内容，或给出文件路径。给路径时用 Read 读取；文件较长时分段读全，不要只读开头就下结论。
- **确认两个关键参数**（用户没说就用默认并说明）：
  - **详略档位**：一句话 / 要点清单 / 结构化摘要（默认按内容长度选：短内容给要点清单，长/复杂内容给结构化摘要）。
  - **用途或读者**：给自己速览、还是要转给别人、还是要留档？用途影响保留哪些细节。

### 步骤 2：判断内容类型

不同类型的内容，"什么是关键信息"完全不同。先判断类型，再决定摘要的骨架：

- **文章 / 报告 / 文档**：关键是**核心论点 + 主要论据 + 结论**。保留作者的主张和支撑逻辑，删掉举例铺垫和重复论证。
- **会议记录 / 纪要**：关键是**决策 + 行动项（谁、做什么、何时）+ 未决问题**。讨论过程可高度压缩，但决定和待办一个都不能漏。
- **代码**：关键是**这段代码做什么、核心逻辑/算法、输入输出、关键依赖和副作用**。不逐行复述，讲清意图和结构；注意隐含的边界条件和风险点。
- **长对话 / 聊天记录**：关键是**达成的共识、分歧点、待办、各方立场**。区分"谁说的"，不要把某个人的观点写成结论。
- **混合内容**：识别主体类型，必要时分段用不同骨架。

### 步骤 3：提取关键信息

- **通读全文再动手**，不要边读边写——摘要需要全局视角才能分清主次。
- 按类型骨架提取：主张/决策/意图对应"主干"，数据/条件/约束对应"关键事实"，待办对应"行动项"。
- **保留具体的关键事实**：数字、日期、名称、金额、指标、明确的结论——这些是最容易在压缩中丢失、也最有价值的信息，优先保留。
- **标注归属**：观点是谁提出的、结论是不是有条件的，都要保持清楚。不要把"A 建议"压缩成"结论是"。
- **区分事实与推测**：原文中的假设、待验证内容，摘要里也要标明是推测而非既定事实。

### 步骤 4：按档位生成摘要

三档由粗到细，按步骤 1 确定的档位输出（也可一次给多档，比如"一句话 + 要点清单"）：

- **一句话摘要**：用一句话说清"这是什么 + 最核心的结论/目的"。适合极速判断。
- **要点清单**：3–8 条 bullet，每条一个关键点，重点在前。适合大多数速览场景。
- **结构化摘要**：分节组织（如：背景 / 核心内容 / 关键结论 / 行动项 / 未决问题），适合长而复杂的内容或需要留档转发的场景。会议纪要和项目文档优先用这一档。

生成通则：

- **重点前置**，最重要的结论放最前面。
- **压缩比合理**：摘要显著短于原文（通常为原文的 5%–20%），但不能短到丢失关键信息。宁可多留一条关键事实，也不为凑字数塞废话。
- **用摘要者的语言重述，不复制原句**（除非是必须精确保留的定义、数据或引文）。避免大段照抄。
- 会议/对话/项目类内容，**行动项单独成节**并尽量补全"谁 / 做什么 / 何时"，缺失的要素标注"（未指定负责人）"。

### 步骤 5：校验与输出

- **保真校验**：逐条对照，摘要里的每个陈述都能在原文找到依据？有没有不小心加入原文没有的推断？有没有把有条件的结论写成绝对结论？
- 默认在对话中输出。用户要求存档时用 Write 保存，文件名建议 `summary-<主题>.md`。

## 输出格式

按所选档位输出。多档时用如下组合：

```markdown
## 一句话
<一句话说清是什么 + 核心结论>

## 要点
- <关键点 1，重点在前>
- <关键点 2>
- ...
```

结构化摘要（长内容 / 会议 / 项目）：

```markdown
# 摘要：<标题或主题>

> 一句话：<核心结论>

## 背景 / 目的
<为什么有这份内容>

## 核心内容
- <主张/决策/逻辑要点>

## 关键结论
- <带数据和条件的结论>

## 行动项
| 事项 | 负责人 | 时间 |
|------|-------|------|

## 未决 / 存疑
- <待验证或未达成一致的点>
```

代码摘要：

```markdown
# 代码摘要：<文件/模块名>

- **作用**：<一句话说清做什么>
- **核心逻辑**：<关键算法/流程>
- **输入 / 输出**：<参数、返回、副作用>
- **依赖 / 注意点**：<关键依赖、边界条件、潜在风险>
```

## 边界

- **不臆造、不添加**：摘要只压缩和重组原文已有信息，绝不补充原文没有的事实、数字或结论。信息不足就如实说"原文未提及"，不脑补。
- **保持归属和条件**：谁说的、在什么前提下成立，压缩后仍要清楚。不把某人观点、有条件结论、推测内容包装成既定事实。
- **不做价值判断**：除非用户明确要求评价，摘要只呈现内容本身，不夹带对内容对错好坏的裁定。
- **关键事实优先于篇幅**：数字、日期、金额、名称、行动项这类高价值信息优先保留，压缩不能以丢失关键事实为代价。
- **敏感/长内容如实处理**：涉及隐私或敏感信息时按原文如实摘要但不额外扩散；内容过长无法一次读完时分段读取，确保覆盖全文再总结，不以偏概全。
- **只做摘要，不做转发**：本技能产出摘要文本，不代替用户发送或发布到任何渠道。
